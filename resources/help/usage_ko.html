TSCRT 사용 설명서

TSCRT는 SSH2 및 시리얼 콘솔용 터미널 에뮬레이터입니다. 주요 기능별 안내는 아래 링크에서 바로 이동할 수 있습니다.

  • 세션(Sessions)
  • 탭 & 멀티 윈도우
  • 분할 창 / 입력 브로드캐스트 / 자동 재접속
  • 버튼(Buttons) — 예제 포함
  • 탭 링크(Tab Link)
  • 스냅샷(Snapshots) — 예제 포함
  • 자동화 규칙(Automation rules)
  • 트리거 / 주기 실행 / 시작 동작
  • SMTP (메일 발송)
  • 로그(Logs)
  • 보안
  • 프로필 가져오기 / 내보내기
  • 업데이트 확인
  • 단축키

세션

세션은 SSH 또는 시리얼 접속 정의 하나를 의미합니다.

  • Sessions → Manage sessions…에서 세션을 추가·수정·복제·삭제합니다. 세션은 왼쪽 도크의 트리에 정리됩니다.
  • SSH: 호스트, 포트, 사용자명, 비밀번호 또는 개인키 파일이 필요합니다.
  • 시리얼: 장치명(예: Windows COM3, macOS /dev/cu.usbserial-xxx), 전송 속도, 데이터/정지/패리티 비트, 흐름 제어가 필요합니다.
  • 왼쪽 도크의 세션을 더블 클릭하면 새 탭으로 엽니다.

세션 편집기 (3탭 구성)

세션 편집 다이얼로그는 세 개의 탭으로 나뉩니다:

  • Connection — 호스트/포트/사용자/비밀번호/키(SSH) 또는 장치/전송속도/라인 설정(Serial). 가장 자주 변경하는 필드가 먼저 보이도록 첫 번째 탭입니다.
  • General — 세션 이름, 접속 유형, 세션별 터미널 타입(예: xterm-256color, vt100), 세션 로깅, 전체화면 표시 옵션. 터미널 타입을 비워 두면 Settings → Preferences의 글로벌 기본값이 적용됩니다.
  • Advanced — 자동 재접속(최대 횟수, 기본 지연), SSH keepalive(간격, TCP keepalive). 시리얼 세션에서는 keepalive 그룹이 숨겨집니다.

탭 & 멀티 윈도우

  • 접속 하나당 탭 하나입니다. 탭 이름 옆의 작은 ×로 해당 탭만 닫을 수 있습니다.
  • Ctrl+Alt+← / → — 이전 / 다음 탭으로 이동합니다.
  • F11 — 전체 화면 토글 (도크와 메뉴 바 숨김). 모든 단축키는 전체화면에서도 동작합니다.

탭 우클릭 메뉴

탭 제목을 우클릭하면 다음 항목을 사용할 수 있습니다:

  • Rename — 탭 표시 이름 변경.
  • Duplicate — 같은 세션을 새 탭으로 열기.
  • Run Snapshot → — 이 탭의 세션에서 스냅샷 실행.
  • Pin / Unpin — 고정된 탭은 다음 실행 시 자동으로 복원됩니다.
  • Detach to New Window — 탭을 새 TSCRT 창으로 분리합니다. SSH/시리얼 연결은 끊기지 않고 그대로 유지됩니다. 탭이 1개일 때는 비활성입니다.
  • Link Tab to Peer Tab… — 이 탭을 같은 머신의 다른 TSCRT 인스턴스에 있는 탭과 페어링합니다. 한쪽 탭을 활성화하면 다른 창의 짝 탭도 함께 전환됩니다. 나란히 배치(예: 왼쪽 모니터에 콘솔, 오른쪽에 로그 뷰어)에 유용합니다.
  • Unlink Tab — 탭에 이미 짝이 있을 때 Link 대신 표시되며, 페어링을 해제합니다.
  • Close — 탭 닫기.

윈도우 간 탭 드래그

탭을 드래그해서 다른 TSCRT 창의 탭 바에 드롭하면 세션이 그 창으로 이동합니다. TSCRT 창 밖으로 드롭하면 새 창이 자동 생성됩니다. 이동 중에도 연결은 끊기지 않습니다.

탭 링크 (Tab Link)

탭 링크는 같은 머신에서 실행 중인 두 TSCRT 창 사이에서 탭 활성화를 동기화하는 기능입니다. 한쪽 창에서 링크된 탭을 클릭하면, 다른 쪽 창의 짝 탭이 자동으로 앞으로 전환됩니다. 듀얼 모니터 환경 — 예: 왼쪽에 장비 콘솔, 오른쪽에 로그 뷰어 또는 다른 장비 — 에 최적화된 기능입니다.

링크 만들기

  1. TSCRT 를 두 번 실행합니다 (첫 번째가 역할 A, 두 번째가 자동으로 역할 B).
  2. 각 창에서 원하는 세션 탭을 엽니다.
  3. 어느 쪽 창에서든 탭 우클릭 → 다른 TSCRT의 탭과 연결…
  4. 대화상자에 상대 TSCRT 인스턴스와 열려 있는 탭 목록이 표시됩니다. 같은 세션 이름의 중복 탭은 슬롯 번호(0, 1, …)로 구분됩니다. 짝지을 탭을 정확히 선택하고 OK.
  5. 링크된 양쪽 탭 제목에 작은 체인 링크 아이콘이 표시되어 연결 상태를 확인할 수 있습니다.

여러 개의 링크를 만들 수 있습니다 — 예: 탭 1↔3, 탭 2↔4 처럼 두 세트의 세션 페어.

활성화 동기화 방식

  • 각 링크에는 고유한 페어 ID가 있습니다. 링크된 탭을 활성화하면 로컬 IPC 채널(Unix 소켓 / Windows 명명된 파이프)을 통해 페어 ID 가 연결된 모든 피어에 전송됩니다.
  • 수신 측은 페어 ID 를 자기 바인딩 테이블에서 찾아 해당 로컬 탭을 활성화합니다. 수신 측 창이 전면으로 올라옵니다.
  • 에코 방지 가드가 있어, 수신에 의한 탭 전환은 다시 송신으로 되돌아가지 않습니다 (무한 루프 차단).
  • 링크는 역할(Role) 기반입니다 — 각 끝점이 어느 프로세스(A 또는 B)에 속하는지 기록하므로, 양쪽 인스턴스에 같은 세션 이름이 있어도 혼동이 발생하지 않습니다.

저장 및 복원

  • 링크는 프로파일에 저장되어 재시작 후에도 유지됩니다. 페어 ID, 양쪽 세션 이름/슬롯/역할이 기록됩니다.
  • 각 프로세스는 현재 열린 탭 목록을 프로파일 디렉터리의 레이아웃 파일(layout_A.lst / layout_B.lst)에 저장합니다.
  • 다음 실행 시, 역할 A 가 저장된 탭을 자동으로 엽니다. 링크가 존재하고 layout_B.lst에 항목이 있으면 두 번째 TSCRT 프로세스(--role=B)가 자동 실행되어 B 측 탭을 열고 모든 링크 바인딩을 재설정합니다 — 클릭 한 번으로 전체 듀얼 윈도우 환경이 복원됩니다.

링크 해제

링크된 탭 우클릭 → 탭 연결 해제 (현재: …). 체인 링크 아이콘이 사라지고 프로파일에서 페어링이 제거됩니다. 모든 링크가 제거되면 다음 실행 시 두 번째 TSCRT 가 자동 생성되지 않습니다.

피어 탐색

각 TSCRT 프로세스는 $XDG_RUNTIME_DIR/tscrt/peers/(Linux) 또는 해당 플랫폼의 임시 디렉터리(Windows/macOS)에 자신을 등록합니다. 3 초마다 주변 피어를 확인합니다. 비정상 종료된 프로세스의 잔여 항목은 소켓 연결 불가 시 자동으로 정리됩니다. 네트워크 트래픽은 발생하지 않습니다.

분할 창 / 입력 브로드캐스트 / 자동 재접속

분할 창 (Split Pane)

한 탭 안에서 터미널을 가로/세로로 분할해 여러 세션을 동시에 운용할 수 있습니다. 분할 시 같은 프로필의 세션이 새 패널에 자동으로 만들어집니다.

  • Ctrl+Shift+H — 가로(좌우) 분할
  • Ctrl+Shift+V — 세로(상하) 분할
  • Ctrl+Shift+W — 활성 패널 닫기. 마지막 패널을 닫으면 탭이 닫힙니다.
  • 분할 트리는 재귀적입니다. 가로 분할 후 한쪽을 다시 세로로 나누면 격자 레이아웃이 됩니다.
  • 활성 패널은 파란색 1px 테두리로 표시됩니다.
  • 패널 사이의 분할선은 마우스로 드래그해 비율을 조정할 수 있습니다.
  • FindBar / Mark / Loop / 명령 입력줄은 항상 활성 패널에 적용됩니다.

입력 브로드캐스트

여러 서버를 동시에 관리할 때 같은 명령을 한 번 타이핑해 모든 패널에 동시에 보내는 기능입니다.

  • View → Broadcast Input 또는 Ctrl+Shift+B로 토글합니다.
  • 활성화되면 현재 탭의 모든 패널이 주황색 테두리로 표시됩니다.
  • 한 패널에서 입력한 키, 붙여넣기, 한글 IME 조합, 마우스 리포트가 다른 모든 패널로 동시에 전송됩니다.
  • 토글은 탭 단위입니다.

자동 재접속 + Keepalive

세션 편집기의 Advanced 탭에서 설정합니다.

  • Auto reconnect — 끊겼을 때 자동 재접속. 사용자가 직접 닫은 탭은 재접속하지 않습니다.
  • Max attempts — 최대 재시도 횟수. 0 = 무제한 (하드 상한 100회).
  • Base delay (ms) — 첫 재시도 대기 (최소 250 ms). 이후 2배씩 증가, 30 s 상한, 0~250 ms 지터.
    예: base 500 → 500 ms → 1 s → 2 s → 4 s → 8 s → 16 s → 30 s …
  • SSH keepalive (s) (SSH 전용) — libssh2 keepalive 간격. 권장 30 s.
  • TCP keepalive (SSH 전용) — TCP 소켓 SO_KEEPALIVE 설정.

재접속 동작: 상태 바에 진행 표시, 스크롤백 유지, 각 패널은 독립적인 재접속 상태를 갖습니다.

버튼

하단 버튼 바에 최대 20개의 사용자 정의 버튼. 클릭하면 전송, 우클릭하면 편집합니다.

예제

1. 명령 전송: uptime\r

2. 시리얼 로그인: \r\padmin\rs3cret\r (Enter → 1초 → admin+Enter → 비밀번호+Enter)

3. Ctrl-C 전송: \x03

4. ESC 후 진단: \e\pshow interfaces\r

Loop / Mark 특수 버튼

  • Loop — 사용자 버튼 우클릭 → Loop this button으로 일정 간격 반복 실행.
  • Mark — 세션 로그에 타임스탬프 마커 삽입. 좌클릭 = 즉시 마크, 우클릭 = 라벨 편집.

스냅샷

스냅샷은 명령 시퀀스에 이름을 붙인 것으로, 세션에서 순서대로 실행한 출력을 캡처하여 파일로 저장합니다. 설정 시 내장 SMTP 클라이언트를 통해 캡처 결과를 지정된 메일 주소로 자동 발송할 수 있습니다.

스냅샷 설정

Snapshots → Manage snapshots…

  • 이름 / 설명
  • Commands — 각 행: 커맨드, 지연(ms), expect prompt(정규식), max wait(ms).
  • Recipients — 스냅샷 결과를 자동으로 받을 이메일 주소 (복수 가능).
  • Subject template — 제목 템플릿: {session}, {snapshot}, {timestamp} 치환 가능.
  • Send email — 스냅샷별 이메일 자동 발송 on/off.
  • Attach file — 켜면 캡처 결과 파일을 이메일에 첨부합니다.

expect_prompt 정규식은 해당 단계 출력의 최근 64 KB 롤링 윈도우에서 매칭됩니다.

이메일 자동 발송

Send email이 켜져 있고 SMTP 가 설정되어 있으면 (SMTP 참조), TSCRT 가 스냅샷 실행 완료 즉시 등록된 모든 수신자에게 결과를 자동 발송합니다. 자동화 규칙(on-connect / cron / pattern)과 결합하면 완전 무인 모니터링이 가능합니다 — TSCRT 가 스케줄 또는 트리거에 맞춰 스냅샷을 실행하고, 출력을 캡처한 뒤, 팀에게 메일로 전달합니다.

실행 방법

  • 메뉴: Snapshots → <스냅샷 이름>, 또는 탭 우클릭 → Run Snapshot.
  • 파일: log dir/snapshots/. Snapshots → Browse snapshots…로 탐색.

자동화 규칙

Snapshots → Automation rules…

  • on_connect — 세션 연결 시 한 번 실행.
  • cron — 5필드 cron 스케줄.
  • pattern — 세션 수신 데이터 정규식 매칭 시 실행. 초 단위 쿨다운.

트리거 / 주기 실행 / 시작 동작

Settings → Preferences에서 설정합니다.

  • Triggers — 세션 출력 패턴 매칭 시 액션 전송. 보안을 위해 rate-limit 적용: 패턴당 2초 쿨다운, 전체 세션당 10초에 최대 20회.
  • Periodics — 지정 세션으로 N초마다 액션 전송.
  • Startup actions — 세션 연결 직후 한 번 액션 전송.

SMTP

Settings → SMTP…

  • Host / Port
  • Security — None, STARTTLS, SMTPS. 사용자명이 설정되어 있으면 STARTTLS 또는 SMTPS 필수 — 평문 인증은 거부됩니다.
  • Username / Password — 비밀번호는 암호화 저장 (Windows: DPAPI, macOS: Keychain).
  • From address / name — Send to administrator 대상 주소로도 사용됩니다.
  • Timeout (s)

TLS 인증서 검증이 강제됩니다: 자가서명, 만료, 호스트명 불일치 인증서는 명확한 에러와 함께 거부됩니다.

로그

  • Logs → Open log folder — 세션별 타임스탬프 파일.
  • 스냅샷 결과: snapshots/ 하위 폴더.
  • 세션별 로그 기록 여부는 세션 편집기 General 탭에서 설정.
  • 자동 폴백: 설정된 로그 디렉터리에 쓸 수 없는 경우(로그인 없이 OneDrive reparse point, Controlled Folder Access, 또는 ACL 제한), 세션 로그는 Windows에서는 C:\TSCRT\sessions\, macOS/Linux에서는 ~/TSCRT/sessions/에 대신 기록됩니다. Open log folder는 이를 감지하여 자동으로 폴백 디렉터리를 열고, 상태 바에 안내를 표시합니다.

보안

SSH 호스트 키 검증

TSCRT는 TOFU(Trust On First Use) 모델을 사용합니다. 서버에 처음 접속하면 호스트 키가 ~/tscrt/known_hosts(OpenSSH 형식)에 고정되고 SHA-256 지문이 기록됩니다.

  • 일치 — 조용히 통과.
  • 신규 호스트 — 자동으로 키 고정 및 저장.
  • 불일치 — 연결 거부. 서버 지문과 known_hosts 경로가 에러에 표시됩니다. 중간자 공격 방어. 서버 키가 정당하게 변경되었다면 known_hosts에서 해당 항목을 제거하고 재접속하세요.

자격 증명 저장

  • Windows — DPAPI 암호화, 현재 사용자 계정에 바인딩. 프로파일 파일에 owner-only ACL 적용.
  • macOS — 시스템 키체인(Security.framework)에 저장. 프로파일에는 keychain:<uuid> 참조만 기록.
  • Linux — 현재 평문 저장 (0600 권한). 향후 libsecret 통합 예정.
  • Windows에서 macOS로 프로파일을 이동하면 암호화된 비밀번호가 풀리지 않습니다. 이동 후 재입력하세요.

크래시 덤프

충돌 시 ~/tscrt/dumps/에 덤프가 기록됩니다 (POSIX: 0600, Windows: owner-only ACL). 7일 이상 된 덤프는 시작 시 자동 삭제됩니다.

지원 & 진단

Help → Export diagnostics…는 시스템 정보, 확인된 경로, 선택된 환경 변수, 쓰기 가능 여부 프로브, 최근 앱 로그 파일(최근 7일), 그리고 크래시 덤프 파일 이름을 포함하는 gzip 텍스트 파일(tscrt-diag-YYYYMMDD-HHMMSS.txt.gz)을 패키징합니다. 일반적인 크기는 1 MB 미만입니다. 압축 전 최대 크기는 10 MB로 제한되며, 필요한 경우 오래된 로그 내용은 명시적인 마커와 함께 잘립니다.

번들을 생성하기 전에 동의 다이얼로그가 정확한 내용을 표시합니다. 번들에는 다음이 포함되지 않습니다: 비밀번호, SSH 개인 키, 세션 프로필(호스트, 스냅샷, 자동화 규칙), 스냅샷 파일, 세션별 터미널 캡처, 또는 사용자가 입력한 명령.

생성된 .txt.gz를 지원팀에 공유하기 전에 검토하세요.

프로필 가져오기 / 내보내기

  • File → Export profile… — 전체 또는 부분.
  • File → Import profile… — 현재 프로필과 병합, 충돌 보고.

업데이트 확인

TSCRT 는 시작 시 GitHub Releases API 를 자동으로 조회하여 새 버전이 있는지 확인합니다. 개인 정보는 전송되지 않으며, 공개 릴리스 엔드포인트에 대한 HTTP GET 요청 한 번만 발생합니다.

동작 방식

  1. 시작 시 api.github.com/repos/nice1972/TSCRT/releases/latest 에 백그라운드 요청 1회.
  2. 응답에서 최신 릴리스 태그(예: v1.0.10) 추출.
  3. 앱에 내장된 현재 버전(TSCRT_VERSION)과 비교.
  4. 원격 버전이 더 높으면 대화상자 표시:
    새 버전 1.0.10 이 있습니다.
    (현재: 1.0.8)

    다운로드 페이지를 여시겠습니까?
  5. Yes 클릭 → 기본 브라우저에서 GitHub Releases 페이지가 열리며 해당 플랫폼의 설치 파일을 받을 수 있습니다.
  6. No 클릭 → 대화상자 닫힘, TSCRT 정상 실행.

업그레이드 방법

플랫폼업그레이드 방법
Ubuntu / Debiansudo apt install ./tscrt_<버전>_amd64.deb — 이전 버전을 자동으로 덮어씁니다.
Windows새 .exe 설치 관리자를 실행하면 기존 설치가 자동 교체됩니다.
macOS새 .dmg 를 열어 tscrt_mac.app 을 /Applications 에 드래그하여 이전 버전을 교체합니다.

프로파일, 세션 로그, 탭 링크 등 ~/.config/tscrt/ (Linux), %APPDATA%\tscrt\ (Windows), ~/Library/Application Support/tscrt/ (macOS) 에 저장된 모든 사용자 데이터는 업그레이드 시 보존됩니다 — 바이너리만 교체됩니다.

오프라인 / 제한 환경

네트워크 요청이 실패하면 (오프라인, DNS 차단, 프록시, 방화벽 등) 업데이트 확인이 자동으로 건너뜁니다. TSCRT 는 응답을 기다리며 시작을 지연시키지 않습니다.
