Sessions

A session is a connection definition — either SSH or serial.

  • Sessions → Manage sessions… — add, edit, duplicate, or delete sessions. Sessions are grouped in a tree in the left dock.
  • SSH requires host, port, username, and either a password or a private-key file.
  • Serial requires a device name (e.g. COM3 on Windows, /dev/cu.usbserial-xxx on macOS), baud rate, data/stop/parity bits, and flow control.
  • Double-click a session in the left dock to open it in a new tab.

Session editor (three tabs)

The session editor is organised into three tabs:

  • Connection — host/port/username/password/key (SSH) or device/baud/line settings (Serial). This tab opens first so the most frequently changed fields are immediately visible.
  • General — session name, connection type, per-session terminal type (e.g. xterm-256color, vt100), session logging toggle, and full-screen display options. If the terminal type is left empty, the global default from Settings → Preferences is used.
  • Advanced — auto-reconnect settings (max attempts, base delay) and SSH keepalive (interval, TCP keepalive). The keepalive group is hidden for serial sessions.

Tabs & multi-window

  • Each open connection lives in its own tab. The tab title shows the session name; a small × next to the name closes just that tab.
  • Ctrl+Alt+← / → — move to the previous / next tab.
  • F11 — toggle fullscreen (hides dock and menu bar). All shortcuts continue to work in fullscreen.

Tab context menu (right-click)

Right-click a tab title to access:

  • Rename — change the displayed tab title.
  • Duplicate — open the same session in a new tab.
  • Run Snapshot → — run a snapshot on this tab's session.
  • Pin / Unpin — pinned tabs are automatically restored on next launch.
  • Detach to New Window — move the tab into a brand-new TSCRT window. The SSH/serial session stays connected — no disconnect/reconnect. Disabled when there is only one tab.
  • Close — close the tab.

Drag tabs between windows

Drag a tab from the tab bar and drop it onto another TSCRT window to transfer the session. If you drop outside any TSCRT window, a new window is automatically created. The active connection is preserved throughout.

Split panes / Input broadcast / Auto-reconnect

Split panes

A single tab can hold several terminal panes arranged in a recursive splitter tree. Splitting reuses the active pane's profile so the new pane lands on the same host.

  • Ctrl+Shift+H — split horizontally (side by side).
  • Ctrl+Shift+V — split vertically (top / bottom).
  • Ctrl+Shift+W — close the active pane. Closing the last pane closes the tab.
  • Splits are recursive: a horizontal split followed by a vertical split produces a grid.
  • The active pane is marked with a 1 px blue border. Mouse clicks and keyboard focus update it automatically.
  • Drag the splitter handle to resize.
  • Find bar, Mark, Loop and the command line always act on the active pane.

Input broadcast

Mirror keystrokes across multiple panes — useful for administering a fleet of servers.

  • Toggle with View → Broadcast Input or Ctrl+Shift+B.
  • While enabled, every pane gets an orange border.
  • Keystrokes, paste, IME input and mouse reports from one pane are forwarded to every other pane.
  • The toggle is per-tab.

Auto-reconnect and SSH keepalive

Configured in the session editor's Advanced tab.

  • Auto reconnect — restore the session after a drop. Manually closed tabs will not reconnect.
  • Max attempts — give-up cap. 0 = unlimited (hard cap at 100).
  • Base delay (ms) — initial backoff (minimum 250 ms). Each retry doubles, ceiling 30 s, with 0–250 ms jitter.
    Example with base 500: 500 ms → 1 s → 2 s → 4 s → 8 s → 16 s → 30 s …
  • SSH keepalive (s) (SSH only) — libssh2 keepalive interval. Recommended 30 s.
  • TCP keepalive (SSH only) — SO_KEEPALIVE on the socket.

During reconnect: the status bar shows progress; scrollback is preserved; each pane has its own independent reconnect state.

Buttons

Up to 20 user-defined action buttons in the bottom bar. Click to send, right-click to edit.

Action string syntax

EscapeMeaning
\rCarriage return (Enter)
\nLine feed
\tTab
\bBackspace
\eEscape (0x1B)
\pPause 1 second
\\Literal backslash

Note: Unknown escape sequences (e.g. \q) are silently dropped and logged as a warning. Only the escapes listed above are processed.

Examples

1. Send a command: uptime\r

2. Serial login: \r\padmin\rs3cret\r (Enter → 1 s → admin+Enter → password+Enter)

3. Send Ctrl-C: \x03

4. ESC then diagnose: \e\pshow interfaces\r

Loop and Mark buttons

  • Loop — right-click a user button → Loop this button to repeat on an interval.
  • Mark — insert a timestamped marker into the session log. Left click = mark, right click = edit label.

Snapshots

A snapshot is a named sequence of commands that runs on a session, captures output, and writes a timestamped file. Optionally emails the result.

Configuring

Snapshots → Manage snapshots…

  • Name / description
  • Commands — each row: command string, delay (ms), optional expect prompt (regex), max wait (ms).
  • Recipients, Subject template ({session}, {snapshot}, {timestamp}).
  • Send email / Attach file.

The expect-prompt regex is matched against a rolling 64 KB window of the step's output, which prevents pathological patterns from stalling the GUI.

Running

  • Menu: Snapshots → <name>, or right-click a tab → Run Snapshot.
  • Files: log dir/snapshots/. Browse via Snapshots → Browse snapshots….

Automation rules

Snapshots → Automation rules…

  • on_connect — run once on session connect.
  • cron — 5-field cron schedule.
  • pattern — regex match on session output, with cooldown (seconds).

Triggers, periodics, startup actions

Configured in Settings → Preferences:

  • Triggers — pattern-match session output and send an action. Rate-limited: each trigger has a 2 s per-pattern cooldown, and a global cap of 20 fires per 10 s across all triggers in a session.
  • Periodics — send an action every N seconds.
  • Startup actions — send an action once on connect.

SMTP

Settings → SMTP…

  • Host / port
  • Security — None, STARTTLS, or SMTPS. If a username is configured, STARTTLS or SMTPS is required — plaintext authentication is refused to prevent credential leakage.
  • Username / password — password is encrypted at rest (DPAPI on Windows, Keychain on macOS).
  • From address / name — also used as the Send to administrator target address.
  • Timeout (s)

TLS certificate validation is enforced: self-signed, expired, or hostname-mismatched certificates are rejected with a clear error message.

Logs

  • Logs → Open log folder — one timestamped file per session.
  • Snapshot captures: snapshots/ subfolder.
  • Per-session logging toggle in the session editor's General tab.

Security

SSH host key verification

TSCRT uses a Trust On First Use (TOFU) model. On the first connection to a server, the host key is pinned to ~/tscrt/known_hosts (OpenSSH format) and its SHA-256 fingerprint is logged.

  • Match — connection proceeds silently.
  • New host — key is automatically pinned and persisted.
  • Mismatch — connection is refused with an error showing the server fingerprint and the known_hosts path. This protects against man-in-the-middle attacks. If the server key legitimately changed, remove the stale entry from the known_hosts file and reconnect.

Credential storage

  • Windows — SSH and SMTP passwords are encrypted with DPAPI, bound to the current user account. The profile file is additionally protected by an owner-only ACL.
  • macOS — passwords are stored in the system Keychain (via Security.framework). The profile contains only a keychain:<uuid> reference.
  • Linux — passwords are currently stored as plaintext in the profile (0600 permissions). A future version may integrate libsecret.
  • Profiles migrated from Windows to macOS will lose encrypted passwords. Re-enter them after migration.

Crash dumps

If TSCRT crashes, a dump is written to ~/tscrt/dumps/ with owner-only permissions (0600 on POSIX, owner-only ACL on Windows). Dumps older than 7 days are automatically deleted on startup.

Profile import / export

  • File → Export profile… — full or partial.
  • File → Import profile… — merges into the current profile; conflicts reported.
