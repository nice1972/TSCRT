TSCRT 使用ガイド

TSCRT は SSH2 およびシリアルコンソール向けのターミナルエミュレーターです。このガイドではすべての主要機能を解説します。セクションへジャンプ:

  • セッション
  • タブとマルチウィンドウ
  • ペイン分割 / 入力ブロードキャスト / 自動再接続
  • ボタン(例付き)
  • タブリンク
  • スナップショット(例付き)
  • 自動化ルール
  • トリガー、定期実行、起動アクション
  • SMTP(メール送信)
  • ログ
  • セキュリティ
  • プロファイルのインポート / エクスポート
  • アップデート確認
  • キーボードショートカット

セッション

セッションとは接続定義のことで、SSH またはシリアルのいずれかです。

  • セッション → セッション管理… — セッションの追加、編集、複製、削除。セッションは左側ドックのツリーでグループ化されます。
  • SSH にはホスト、ポート、ユーザー名、およびパスワードまたは秘密鍵ファイルが必要です。
  • シリアル にはデバイス名(例: Windows では COM3、macOS では /dev/cu.usbserial-xxx)、ボーレート、データ/ストップ/パリティビット、フロー制御が必要です。
  • 左側ドックでセッションをダブルクリックすると、新しいタブで開きます。

セッションエディター(3 つのタブ)

セッションエディターは 3 つのタブに分かれています:

  • 接続 — ホスト/ポート/ユーザー名/パスワード/鍵(SSH)またはデバイス/ボーレート/ライン設定(シリアル)。最も頻繁に変更されるフィールドをすぐに表示するため、このタブが最初に開きます。
  • 一般 — セッション名、接続タイプ、セッションごとのターミナルタイプ(例: xterm-256color、vt100)、セッションログの切り替え、フルスクリーン表示オプション。ターミナルタイプを空のままにすると、設定 → 環境設定のグローバルデフォルトが使用されます。
  • 詳細設定 — 自動再接続設定(最大試行回数、基本遅延)と SSH キープアライブ(間隔、TCP キープアライブ)。キープアライブグループはシリアルセッションでは非表示になります。

タブとマルチウィンドウ

  • 開かれた各接続は独自のタブに存在します。タブタイトルはセッション名を表示し、名前の横の小さな × でそのタブだけを閉じます。
  • Ctrl+Alt+← / → — 前 / 次のタブへ移動。
  • F11 — フルスクリーンの切り替え(ドックとメニューバーを非表示)。すべてのショートカットはフルスクリーンでも動作します。

タブのコンテキストメニュー(右クリック)

タブタイトルを右クリックすると、以下にアクセスできます:

  • 名前の変更 — 表示されるタブタイトルを変更。
  • 複製 — 同じセッションを新しいタブで開く。
  • スナップショットを実行 → — このタブのセッションでスナップショットを実行。
  • 固定 / 固定解除 — 固定されたタブは次回起動時に自動的に復元されます。
  • 新しいウィンドウに切り離す — タブを全く新しい TSCRT ウィンドウに移動。SSH/シリアルセッションは接続されたまま — 切断/再接続はありません。タブが 1 つしかない場合は無効です。
  • ピアタブにリンク… — このタブを同じマシン上の別の TSCRT インスタンスのタブとペアリングします。どちらかのタブをアクティブにすると、もう一方のウィンドウのペアタブも同時に切り替わります。左右に並べたレイアウト(例: 左モニターにコンソール、右にログビューア)に便利です。
  • タブのリンク解除 — タブにすでにペアがある場合、リンクの代わりに表示され、ペアリングを解除します。
  • 閉じる — タブを閉じる。

ウィンドウ間でのタブのドラッグ

タブバーからタブをドラッグし、別の TSCRT ウィンドウにドロップしてセッションを移動します。TSCRT ウィンドウ外にドロップすると、新しいウィンドウが自動的に作成されます。アクティブな接続は終始保持されます。

タブリンク

タブリンクは、同じマシン上で実行されている 2 つの TSCRT ウィンドウ間でタブのアクティブ化を同期します。一方のウィンドウでリンクされたタブに切り替えると、もう一方のウィンドウのペアタブが自動的に前面に表示されます。これはデュアルモニターワークフロー向けに設計されています — たとえば、左画面にデバイスコンソール、右画面にログビューアまたは 2 台目のデバイスを配置する場合です。

リンクの作成

  1. 2 つの TSCRT インスタンスを開きます(最初がロール A、2 番目が自動的にロール Bになります)。
  2. 各ウィンドウで目的のセッションタブを開きます。
  3. いずれかのウィンドウでタブを右クリック → ピアタブにリンク…
  4. ダイアログに、もう一方の TSCRT インスタンスとそのオープンタブが一覧表示されます。同じセッション名のタブはスロット番号(0、1、…)で区別されます。ペアリングしたい正確なピアタブを選択し、OK をクリックします。
  5. リンクされた両方のタブに小さなチェーンリンクアイコンが表示され、ペアリングが確認されます。

複数のリンクを作成できます — たとえば、タブ 1↔3 とタブ 2↔4 をリンクして、関連するセッションの 2 つのペアを作成します。

アクティブ化同期の仕組み

  • 各リンクは一意のペア ID で識別されます。リンクされたタブをアクティブにすると、TSCRT はローカル IPC チャネル(Unix ソケット / Windows 名前付きパイプ)を介してペア ID を接続されたすべてのピアにブロードキャストします。
  • 受信側のピアはバインディングテーブルでペア ID を検索し、一致するローカルタブを見つけて自動的に切り替えます。ピアウィンドウも前面に表示されます。
  • ループ防止ガードにより、アクティブ化がバウンスしないようにします。受信メッセージによってタブが切り替えられた場合、送信ブロードキャストは一時的に抑制されます。
  • リンクはロール対応です — 各エンドポイントは自分がどのプロセスロール(A または B)に属するかを記録するため、インスタンス間で重複するセッション名が曖昧なバインディングを引き起こすことはありません。

保存と復元

  • リンクはプロファイルに保存され、再起動後も維持されます。各リンクはペア ID、両方のエンドポイントのセッション名、スロットインデックス、プロセスロールを記録します。
  • 各プロセスはオープンタブリストもプロファイルディレクトリのレイアウトファイル(layout_A.lst / layout_B.lst)に保存します。
  • 次回起動時、ロール A は保存されたタブを自動的に再度開きます。リンクが存在し layout_B.lst が空でない場合、2 番目の TSCRT プロセス(--role=B)が自動的にスポーンされ、B 側のタブを開き、すべてのリンクバインディングを再確立します — デュアルウィンドウ環境全体がワンクリックで復元されます。

リンク解除

リンクされたタブを右クリック → タブのリンク解除(現在: …)。チェーンリンクアイコンが消え、ペアリングがプロファイルから削除されます。すべてのリンクが削除されると、次回の起動で 2 番目の TSCRT が自動的にスポーンされなくなります。

ピア検出

各 TSCRT プロセスは $XDG_RUNTIME_DIR/tscrt/peers/(Linux)または同等の一時ディレクトリ(Windows / macOS)に登録されます。軽量の検出タイマーが 3 秒ごとに新しいピアをチェックします。クラッシュしたプロセスの古いエントリは、ソケットに到達できないことが判明すると自動的にクリーンアップされます。ネットワークトラフィックがローカルマシンを離れることはありません。

ペイン分割 / 入力ブロードキャスト / 自動再接続

ペイン分割

1 つのタブには、再帰的なスプリッターツリーに配置された複数のターミナルペインを含めることができます。分割はアクティブペインのプロファイルを再利用するため、新しいペインは同じホスト上に開きます。

  • Ctrl+Shift+H — 水平分割(左右に並べる)。
  • Ctrl+Shift+V — 垂直分割(上下に並べる)。
  • Ctrl+Shift+W — アクティブペインを閉じる。最後のペインを閉じるとタブが閉じます。
  • 分割は再帰的です: 水平分割の後に垂直分割を行うとグリッドが生成されます。
  • アクティブペインは 1 px の青い枠線 でマークされます。マウスクリックとキーボードフォーカスにより自動更新されます。
  • スプリッターハンドルをドラッグしてサイズ変更します。
  • 検索バー、マーク、ループ、コマンドラインは常にアクティブペインに作用します。

入力ブロードキャスト

複数のペインにキー入力をミラーリング — サーバー群の管理に便利です。

  • 表示 → 入力をブロードキャスト または Ctrl+Shift+B で切り替え。
  • 有効な間、各ペインには オレンジの枠線 が表示されます。
  • あるペインからのキー入力、貼り付け、IME 入力、マウス報告は他のすべてのペインに転送されます。
  • 切り替えはタブごとです。

自動再接続と SSH キープアライブ

セッションエディターの詳細設定タブで構成します。

  • 自動再接続 — 切断後にセッションを復元します。手動で閉じられたタブは再接続されません。
  • 最大試行回数 — 断念の上限。0 = 無制限(ハード上限 100)。
  • 基本遅延(ms) — 初期バックオフ(最小 250 ms)。各再試行で倍増し、上限 30 s、0–250 ms のジッターがあります。
    基本 500 の例: 500 ms → 1 s → 2 s → 4 s → 8 s → 16 s → 30 s …
  • SSH キープアライブ(s)(SSH のみ) — libssh2 のキープアライブ間隔。推奨は 30 s。
  • TCP キープアライブ(SSH のみ) — ソケットの SO_KEEPALIVE。

再接続中: ステータスバーに進捗が表示されます。スクロールバックは保持されます。各ペインは独自の独立した再接続状態を持ちます。

ボタン

下部バーには最大 20 個のユーザー定義アクションボタンがあります。クリックで送信、右クリックで編集。

アクション文字列の構文

エスケープ意味
\rキャリッジリターン(Enter)
\nラインフィード
\tタブ
\bバックスペース
\eエスケープ(0x1B)
\p1 秒間のポーズ
\\リテラルのバックスラッシュ

注: 不明なエスケープシーケンス(例: \q)は静かに破棄され、警告としてログに記録されます。上記のエスケープのみが処理されます。

例

1. コマンドを送信: uptime\r

2. シリアルログイン: \r\padmin\rs3cret\r(Enter → 1 s → admin+Enter → password+Enter)

3. Ctrl-C を送信: \x03

4. ESC の後に診断: \e\pshow interfaces\r

ループボタンとマークボタン

  • ループ — ユーザーボタンを右クリック → このボタンをループで、一定間隔で繰り返します。
  • マーク — タイムスタンプ付きマーカーをセッションログに挿入。左クリック = マーク、右クリック = ラベル編集。

スナップショット

スナップショットとは、セッション上で実行される名前付きのコマンド列で、出力をキャプチャしタイムスタンプ付きファイルに書き込みます。設定されている場合、TSCRT は内蔵の SMTP クライアントを使用して、キャプチャした結果を自動的にメールで送信します。

構成

スナップショット → スナップショット管理…

  • 名前 / 説明
  • コマンド — 各行: コマンド文字列、遅延(ms)、オプションの期待プロンプト(正規表現)、最大待機時間(ms)。
  • 受信者 — スナップショット結果を自動的に受信する 1 つ以上のメールアドレス。
  • 件名テンプレート — プレースホルダーをサポート: {session}、{snapshot}、{timestamp}。
  • メール送信 — スナップショットごとの自動メール配信のオン/オフを切り替えます。
  • ファイル添付 — 有効にすると、キャプチャした出力ファイルがメールに添付されます。

期待プロンプトの正規表現は、ステップの出力のローリング 64 KB ウィンドウに対して照合されるため、病的なパターンが GUI を停止させることを防ぎます。

自動メール配信

スナップショットでメール送信が有効で、SMTP が設定されている場合(SMTP を参照)、TSCRT は実行完了直後にスナップショット結果をリストされたすべての受信者に自動的に配信します。自動化ルール(on_connect / cron / pattern)と組み合わせることで、完全に無人の監視が実現します: TSCRT はスケジュールまたはトリガーに基づいてスナップショットを実行し、出力をキャプチャし、チームにメールで送信します — 手動介入は不要です。

実行

  • メニュー: スナップショット → <名前>、またはタブを右クリック → スナップショットを実行。
  • ファイル: ログディレクトリ/snapshots/。スナップショット → スナップショットを参照… から参照。

自動化ルール

スナップショット → 自動化ルール…

  • on_connect — セッション接続時に一度実行。
  • cron — 5 フィールドの cron スケジュール。
  • pattern — セッション出力に対する正規表現一致、クールダウン(秒)付き。

トリガー、定期実行、起動アクション

設定 → 環境設定で構成:

  • トリガー — セッション出力のパターンに一致させてアクションを送信。レート制限あり: 各トリガーはパターンあたり 2 秒のクールダウン、セッション内のすべてのトリガー合計で 10 秒あたり 20 回の発火というグローバル上限があります。
  • 定期実行 — N 秒ごとにアクションを送信。
  • 起動アクション — 接続時に一度アクションを送信。

SMTP

設定 → SMTP…

  • ホスト / ポート
  • セキュリティ — なし、STARTTLS、または SMTPS。ユーザー名が構成されている場合、STARTTLS または SMTPS が必要です — 資格情報漏洩を防ぐため、プレーンテキスト認証は拒否されます。
  • ユーザー名 / パスワード — パスワードは保存時に暗号化されます(Windows では DPAPI、macOS では Keychain)。
  • 送信者アドレス / 名前 — 管理者に送信の宛先アドレスとしても使用されます。
  • タイムアウト(s)

TLS 証明書検証が強制されます: 自己署名、期限切れ、またはホスト名が一致しない証明書は明確なエラーメッセージとともに拒否されます。

ログ

  • ログ → ログフォルダを開く — セッションごとに 1 つのタイムスタンプ付きファイル。
  • スナップショットキャプチャ: snapshots/ サブフォルダ。
  • セッションごとのログ切り替えは、セッションエディターの一般タブにあります。
  • 自動フォールバック: 設定したログディレクトリに書き込めない場合(サインインなしの OneDrive reparse point、Controlled Folder Access、または ACL 制限)、セッションログは代わりに Windows では C:\TSCRT\sessions\、macOS/Linux では ~/TSCRT/sessions/ に書き込まれます。ログフォルダを開くはこれを検出し、フォールバックディレクトリを自動的に開き、ステータスバーに通知を表示します。

セキュリティ

SSH ホスト鍵の検証

TSCRT は Trust On First Use (TOFU) モデルを使用します。サーバーへの最初の接続時に、ホスト鍵は ~/tscrt/known_hosts(OpenSSH 形式)に固定され、その SHA-256 フィンガープリントがログに記録されます。

  • 一致 — 接続は静かに進行します。
  • 新規ホスト — 鍵は自動的に固定され永続化されます。
  • 不一致 — 接続は拒否され、サーバーのフィンガープリントと known_hosts のパスを示すエラーが表示されます。これは中間者攻撃から保護します。サーバーの鍵が正当に変更された場合は、known_hosts ファイルから古いエントリを削除して再接続してください。

資格情報の保管

  • Windows — SSH および SMTP パスワードは現在のユーザーアカウントに紐付けられた DPAPI で暗号化されます。プロファイルファイルは所有者のみの ACL でさらに保護されます。
  • macOS — パスワードはシステム Keychain に保存されます(Security.framework 経由)。プロファイルには keychain:<uuid> 参照のみが含まれます。
  • Linux — パスワードは現在プロファイル内にプレーンテキストで保存されます(0600 権限)。将来のバージョンで libsecret と統合される可能性があります。
  • Windows から macOS に移行されたプロファイルは暗号化されたパスワードを失います。移行後に再入力してください。

クラッシュダンプ

TSCRT がクラッシュした場合、ダンプは所有者のみのアクセス権(POSIX では 0600、Windows では所有者のみの ACL)で ~/tscrt/dumps/ に書き込まれます。7 日以上古いダンプは起動時に自動的に削除されます。

サポート & 診断

ヘルプ → 診断をエクスポート… は、システム情報、解決済みのパス、選択された環境変数、書き込み可能性プローブ、最近のアプリログファイル(過去 7 日間)、およびクラッシュダンプの名前を含む gzip 圧縮テキストファイル(tscrt-diag-YYYYMMDD-HHMMSS.txt.gz)をパッケージ化します。一般的なサイズは 1 MB 未満です。圧縮前の最大サイズは 10 MB に制限されており、必要に応じて古いログ内容は明示的なマーカー付きで切り詰められます。

バンドルが書き込まれる前に同意ダイアログが正確な内容を表示します。バンドルには次のものは含まれません: パスワード、SSH 秘密鍵、セッションプロファイル(ホスト、スナップショット、自動化ルール)、スナップショットファイル、セッションごとのターミナルキャプチャ、または入力したコマンド。

生成された .txt.gz をサポートに共有する前に確認してください。

プロファイルのインポート / エクスポート

  • ファイル → プロファイルをエクスポート… — 完全または部分。
  • ファイル → プロファイルをインポート… — 現在のプロファイルにマージされ、競合が報告されます。

アップデート確認

TSCRT は起動時に GitHub Releases API を照会して、新しいバージョンを自動的にチェックします。個人データは送信されません — 公開リリースエンドポイントへの単一の HTTP GET リクエストのみです。

動作の仕組み

  1. 起動時、TSCRT は api.github.com/repos/nice1972/TSCRT/releases/latest にバックグラウンドリクエストを送信します。
  2. レスポンスには最新のリリースタグ(例: v1.0.10)が含まれます。
  3. TSCRT はこれをコンパイル時に組み込まれたバージョン(TSCRT_VERSION)と比較します。
  4. リモートバージョンがより新しい場合、ダイアログが表示されます:
    TSCRT の新しいバージョンが利用可能です: 1.0.10
    (現在: 1.0.8)

    ダウンロードページを開きますか?
  5. はいをクリックすると、デフォルトブラウザで GitHub Releases ページが開き、お使いのプラットフォーム用のインストーラーをダウンロードできます。
  6. いいえをクリックするとダイアログが閉じ、TSCRT は通常どおり続行します。

アップグレード方法

プラットフォームアップグレード手順
Ubuntu / Debiansudo apt install ./tscrt_<version>_amd64.deb — 以前のバージョンを上書きします。
Windows新しい .exe インストーラーを実行します。既存のインストールが自動的に置き換えられます。
macOS新しい .dmg を開き、tscrt_mac.app を /Applications にドラッグして、古いバージョンを置き換えます。

プロファイル、セッションログ、タブリンク、および ~/.config/tscrt/(Linux)、%APPDATA%\tscrt\(Windows)、または ~/Library/Application Support/tscrt/(macOS)に保存されたすべてのユーザーデータは、アップグレード後も保持されます — バイナリのみが置き換えられます。

オフライン / 制限された環境

ネットワークリクエストが失敗した場合(オフライン、DNS ブロック、プロキシ、ファイアウォール)、チェックは静かにスキップされます。TSCRT はレスポンスを待って起動をブロックすることはありません。
